# CoPay

Веб-приложение для **совместного учёта расходов**: разовые расчёты без регистрации у всех участников и **комнаты** для постоянных групп с аккаунтами, балансами и приглашениями.

**Стек:** Django 5 (REST JSON API) + React 18 (Vite, TypeScript) + JWT. Статика фронта отдаётся отдельно или через прокси к API.

## Функциональность

### Без аккаунта (гости)

- **Разовый расчёт** (`/calculator`): ввод участников, позиций чека с указанием, кто в чём участвовал; выбор плательщика и валюты (RUB / USD / EUR); расчёт переводов и **публичная ссылка** на результат (`/r/:id`).
- **Просмотр результата** по ссылке: список позиций, доли по людям, упрощённые переводы, блок «мои переводы» при вводе имени как у участника.
- **Демо разбора чека**: загрузка файла → тестовый ответ сервера (заглушка под OCR).

### С аккаунтом

- **Регистрация / вход** по email или телефону и паролю.
- **Профиль**: имя, фамилия, email (при сохранении email синхронизируется с логином на бэкенде), телефон, **код приглашения** в комнату, **аватар** (загрузка изображения).
- **Комнаты**: создание с названием и **валютой комнаты**; список своих комнат; переход в комнату.
- **Комната**:
  - участники и **личные переводы** (кому вы должны / кто должен вам) в валюте комнаты;
  - **история расходов** с суммой, плательщиком, при необходимости списком позиций;
  - **создание / редактирование / удаление** расхода (в т.ч. режим «позиции чека» с разными участниками на строку или одна сумма поровну на всех);
  - **приглашение** (только создатель): поиск пользователей по ФИО/логину или по **коду из профиля** (модальное окно).
- **Настройки**: язык интерфейса (RU/EN), светлая/тёмная тема, цвет акцента; часть настроек хранится в профиле пользователя в БД.
- **Контакты**: страница с email для обратной связи.

### Общее (UI)

- Шапка: калькулятор, комнаты, контакты; для авторизованных — меню по аватару (профиль, настройки, выход).
- Клавиатура: стрелки на главной карусели; Escape в модальных окнах.
- Строки интерфейса подгружаются с API (`/api/i18n`) и дополняются встроенными fallback-словарями на фронте.

## Сценарии использования

### 1. Поездка / ужин: один раз посчитать, без регистрации друзей

1. Открыть **«Расчёт»** (`/calculator`).
2. Добавить имена участников.
3. Для каждой позиции чека указать название, сумму и отметить, кто ею пользовался.
4. Выбрать, **кто оплатил**, валюту.
5. Нажать расчёт → открывается страница результата с **ссылкой**, которой можно поделиться.
6. Каждый может открыть ссылку и в блоке «мои переводы» ввести **своё имя** как в списке — увидеть, кому переводить и кто ему должен.

### 2. Съёмная квартира / семья: постоянный учёт

1. Зарегистрироваться и войти.
2. В **«Комнаты»** создать комнату (название + валюта).
3. В комнате создатель нажимает **«Добавить людей»**: ищет соседей по имени или вводит их **код приглашения** из профиля.
4. Участники добавляют **расходы** (с детализацией по позициям или одной суммой на всех).
5. В правой колонке смотрят **личный баланс** (кому скинуть / кто им скинет); при необходимости раскрывают полный список переводов по всем.

### 3. Исправить ошибку в расходе

1. В комнате в истории у строки нажать **«Изменить»** → модальное окно с теми же режимами (позиции / одна сумма).
2. Сохранить или удалить расход.

### 4. Смена языка и оформления

1. Меню аватара → **Настройки**.
2. Выбрать язык, тему, акцент → **Сохранить** (синхронизация с `/api/me` и локальным хранилищем для первого отображения).

### 5. Вход только по почте после смены в профиле

1. В профиле указать **email** и сохранить — на сервере логин (`username`) привязывается к этому email (если нет конфликта).
2. Дальше входить можно с этим email и паролем.

## Маршруты фронтенда

| Путь | Назначение |
|------|------------|
| `/` | Главная, описание возможностей, карусель |
| `/calculator` | Разовый расчёт |
| `/r/:id` | Результат разового расчёта по публичному id |
| `/rooms` | Список комнат / создание (нужен вход) |
| `/rooms/:roomId` | Комната: баланс, участники, история, расходы |
| `/contacts` | Контакты |
| `/login`, `/register` | Вход и регистрация |
| `/profile` | Профиль и код приглашения |
| `/settings` | Язык, тема, акцент |

В режиме разработки Vite проксирует `/api` и `/media` на бэкенд (см. `frontend/vite.config.ts`).

## HTTP API

Базовый префикс: **`/api/`**. Ответы — JSON. Ошибки: как правило `{ "error": "текст" }` и HTTP-код 4xx/5xx.

### Система

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/api/health` | Проверка живости сервиса |

### Регистрация и вход

| Метод | Путь | Тело (JSON) | Ответ |
|--------|------|-------------|--------|
| POST | `/api/auth/register` | `email` или используется телефон из поля логина, `password`, `displayName`, опционально `lastName`, `phone` | `201` — `token`, `user` |
| POST | `/api/auth/login` | `email` **или** `phone` (одно из полей, как на фронте), `password` | `token`, `user` |

### Текущий пользователь (JWT)

Заголовок: `Authorization: Bearer <token>`.

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/api/me` | Профиль: имя, логин, email, телефон, `inviteCode`, `avatarUrl`, `preferredLanguage`, `theme`, `accent` |
| PATCH | `/api/me` | Частичное обновление: `displayName`, `lastName`, `email`, `phone`, `login` (опционально), `preferredLanguage`, `theme`, `accent`. При заданном **email** логин может быть выровнен под email (см. логику на сервере) |
| POST | `/api/me/avatar` | `multipart/form-data`, поле файла `file` или `avatar` (изображение, до ~5 МБ) | `avatarUrl` |

### Интернационализация

| Метод | Путь | Параметры | Ответ |
|--------|------|-----------|--------|
| GET | `/api/i18n` | `lang=ru` или `en` | Объект `{ "ключ": "перевод" }` из таблицы `TextResource` |

### Поиск пользователей (для приглашений)

| Метод | Путь | Параметры | Auth |
|--------|------|-----------|------|
| GET | `/api/users/search` | `q` — строка поиска (от 2 символов) | JWT |

### Комнаты

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/api/rooms` | Список комнат текущего пользователя |
| POST | `/api/rooms` | Создание: `name` (≥2 символов), `currency` — `RUB` / `USD` / `EUR` (по умолчанию RUB) |
| GET | `/api/rooms/<room_id>` | Детали: участники, валюта, создатель |
| POST | `/api/rooms/<room_id>/members` | **Только создатель.** Тело: `userId` **или** `inviteCode` — добавить участника |

### Расходы в комнате

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/api/rooms/<room_id>/expenses` | Список расходов |
| POST | `/api/rooms/<room_id>/expenses` | Создание. Варианты тела: **A)** `lineItems`: `[{ "name", "amount", "participantIds": [uuid, ...] }]` — суммы и доли считаются из позиций; **B)** `title`, `amount`, `payerId`, опционально `shares` (веса по `member_id`); плательщик должен быть участником |
| GET | `/api/rooms/<room_id>/expenses/<expense_id>` | Один расход |
| PATCH | `/api/rooms/<room_id>/expenses/<expense_id>` | Обновление (те же правила, что у POST) |
| DELETE | `/api/rooms/<room_id>/expenses/<expense_id>` | Удаление |

### Баланс и переводы в комнате

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/api/rooms/<room_id>/balance` | `currency`, `balances` (по отображаемым именам), `transfers` (упрощённые долги), `viewer` — для **текущего пользователя** `payTo[]`, `receiveFrom[]`, `perMember` — то же по всем именам |

### Разовый расчёт (adhoc)

| Метод | Путь | Описание |
|--------|------|----------|
| POST | `/api/adhoc/split` | **Режим 1:** `payer`, `currency`, `products`: `[{ "name", "price", "participants": ["имя", ...] }]` — итог и доли из позиций. **Режим 2 (legacy):** `payer`, `amount`, `participants` — равные доли между всеми. Ответ: `id` (публичный), `transfers`, `shareLink`, `currency` и др. |
| GET | `/api/adhoc/<public_id>` | Сохранённый расчёт. Опционально `?viewerName=Имя` — блок `viewer` с переводами от лица этого имени |

### Чеки (демо)

| Метод | Путь | Тело | Описание |
|--------|------|------|----------|
| POST | `/api/receipts/mock-parse` | `{ "fileName": "..." }` | Заглушка под OCR, тестовый `note` / `items` |

### Статика медиа

| Метод | Путь | Описание |
|--------|------|----------|
| GET | `/media/...` | Файлы из `MEDIA_ROOT` (аватары и т.д.) |

## Аутентификация

- После `register` / `login` клиент сохраняет **JWT** и передаёт его в заголовке `Authorization: Bearer ...` для защищённых эндпоинтов.
- Эндпоинты без токена: `health`, `auth/*`, `i18n`, `adhoc/split`, `adhoc/<id>`, `receipts/mock-parse`, раздача `/media/`.

## Локальный запуск

### Бэкенд

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate   # Windows
pip install -r requirements.txt
python manage.py migrate
python manage.py runserver 0.0.0.0:8000
```

Зависимости см. `backend/requirements.txt` (Django, PyJWT, Pillow, django-cors-headers, gunicorn).

### Фронтенд

```bash
cd frontend
npm install
npm run dev
```

По умолчанию Vite поднимает dev-сервер и проксирует `/api` и `/media` на `http://127.0.0.1:8000`.

Сборка продакшена:

```bash
npm run build
```

## Медиа и аватары

- Загруженные аватары сохраняются под управлением Django (`MEDIA_ROOT`, URL `/media/`).
- Фронтенд может нормализовать абсолютные URL к пути `/media/...`, чтобы в разработке запросы шли через тот же origin, что и SPA (прокси).

---

## Интернационализация и темы

- Переводы: модель **`TextResource`** в БД (`key`, `ru`, `en`), выдача через **`GET /api/i18n?lang=`**; на клиенте есть запасные строки на RU/EN до загрузки и для отсутствующих ключей.
- Тема и акцент задаются атрибутами на `<html>` (`data-theme`, `data-accent`) и в профиле пользователя (`theme`, `accent`).
